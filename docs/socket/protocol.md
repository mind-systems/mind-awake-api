# WebSocket — Протокол

Клиент подключается к двум пространствам имён одновременно: `/live` для событий активности и управления присутствием, `/telemetry` для потоковой передачи биосигналов и других высокочастотных данных. Оба пространства используют JWT-аутентификацию: токен передаётся в поле `auth.token` при handshake. Соединение будет отклонено, если токен отсутствует, истёк или не найден в таблице `user_sessions`.

## Конверт сообщения

Каждое сообщение оборачивается в версионированный конверт. Поле `v` содержит версию протокола (сейчас `1`) и позволяет серверу и клиенту плавно переходить на новую схему. Поле `id` — уникальный UUID сообщения; он используется для дедупликации и сопоставления в логах. Поле `timestamp` — клиентское unix-время в миллисекундах, необходимое для измерения задержки и правильного упорядочивания событий. Поле `type` содержит имя события, `payload` — произвольный объект с данными события.

## Пространство имён `/live`

Через `/live` проходят события управления сессией и присутствием.

**События от клиента к серверу:**

| Событие | Описание |
|---------|----------|
| `activity:start` | Начало отслеживаемой сессии. Payload содержит `activityType` (например `breathing`), и опциональную пару `activityRefType` / `activityRefId` для привязки к конкретной записи в базе без жёсткого FK-ограничения. |
| `activity:end` | Необязательный сигнал чистого завершения. Содержит `durationSeconds`. Сервер завершает сессию со статусом `completed` не дожидаясь разрыва соединения. |
| `presence:background` | Приложение ушло в фон. Сервер обновляет статус пользователя. |
| `presence:foreground` | Приложение вернулось на экран. |

**События от сервера к клиенту:**

| Событие | Описание |
|---------|----------|
| `session:state` | Отправляется при подключении и после `activity:start`. Payload: `liveSessionId`, `status`, `startedAt`, `resumed`. Если при подключении сервер нашёл незавершённую сессию в grace-периоде — `resumed` будет `true`. |
| `session:error` | Ошибка протокола или аутентификации. Содержит `code` и человекочитаемое `message`. |

## Пространство имён `/telemetry`

Через `/telemetry` передаются высокочастотные данные — биосигналы, метрики дыхания и прочая потоковая телеметрия. Это пространство имён намеренно отделено от `/live`, чтобы высокочастотный трафик не мешал управляющим событиям.

**События от клиента к серверу:**

| Событие | Описание |
|---------|----------|
| `data:stream` | Батч сэмплов телеметрии. Payload: `liveSessionId`, `dataType`, `samples` — массив объектов с данными. Максимальный размер одного сообщения ограничен 64 КБ; сообщения, превышающие лимит, отклоняются без обработки. |

**События от сервера к клиенту:**

| Событие | Описание |
|---------|----------|
| `data:ack` | Подтверждение получения батча. Содержит `receivedCount` — количество принятых сэмплов, `droppedCount` — количество сброшенных из-за переполнения буфера, и `maxSamplesPerSecond` — рекомендуемый лимит частоты отправки. Клиент обязан учитывать эту подсказку и самостоятельно замедляться при необходимости. |

## Валидация входящих сообщений

Все входящие события проходят через pipeline валидации на основе `class-validator` до того, как попасть в бизнес-логику. Сообщения с неверной схемой или превышающие лимит размера отклоняются с ответом `session:error` — они не достигают `ActivityEngine` или `StreamEngine`.
