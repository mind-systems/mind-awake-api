# Email Authentication

Аутентификация построена на одноразовых кодах — пользователь не задаёт пароль, вместо этого при каждом входе на указанный адрес приходит письмо с шестизначным кодом.

## Как устроен вход

Клиент начинает с `POST /auth/send-code`, передав только email. Сервер проверяет, не было ли уже отправлено письмо за последние 60 секунд: если было — возвращает 429, не раскрывая никаких деталей о том, зарегистрирован ли этот адрес. Если лимит не превышен, старые коды для этого email немедленно удаляются, генерируется новый шестизначный код через `crypto.randomInt` (криптографически стойкий RNG), хешируется SHA-256 и сохраняется в таблице `auth_codes` со временем жизни 15 минут. Всё это происходит внутри одной транзакции с пессимистичной блокировкой, чтобы параллельные запросы не могли создать два кода одновременно. Только после успешного коммита транзакции сервер отправляет письмо через Resend; если отправка упала — запись из базы откатывается вручную, и клиент получает ошибку.

Ответ `POST /auth/send-code` всегда одинаковый: `{ "message": "If this email is registered, a code has been sent." }` — намеренно обезличенный, чтобы нельзя было перебором выяснить, существует ли аккаунт.

Когда пользователь видит код в письме (или нажимает magic link с ним), клиент отправляет `POST /auth/verify-code` с полями email, сам код и опциональным `language`. Сервер хеширует код, делает SELECT с пессимистичной блокировкой (`FOR UPDATE`) по тройному условию — хеш, email и `used = false` и `expires_at > now()`. Без совмещения email с хешем украденный код можно было бы использовать против любого другого адреса; проверка email закрывает этот вектор. Если строка найдена, она сразу помечается как использованная (`used = true`) в той же транзакции — повторное предъявление кода ничего не даст даже при гонке запросов. Затем сервер ищет пользователя по email: если его нет — создаёт автоматически, взяв часть адреса до `@` как имя и сохранив `language` из запроса как предпочтительный язык интерфейса. На выходе формируется JWT, который возвращается одновременно в теле ответа (`{ "accessToken": "..." }`) и в заголовке `Authorization: Bearer <token>`.

## Хранение кодов

Таблица `auth_codes` хранит email, SHA-256 хеш кода, время создания, время истечения и флаг использования. Сырой код никогда не попадает в базу. Три индекса — по `code_hash`, `email` и `expires_at` — обеспечивают быстрый поиск при верификации, проверку cooldown при отправке и эффективную очистку устаревших строк. Раз в час срабатывает cron-задача, которая удаляет все записи с `expires_at < now()`.

## Письмо

Письма отправляются через [Resend](https://resend.com). Сервис инициализируется при старте приложения; если переменная `RESEND_API_KEY` не задана — старт падает с ошибкой, не позволяя запустить сервер без рабочей почты.

Письмо локализовано: при отправке кода клиент передаёт опциональное поле `locale` в теле `POST /auth/send-code` (например `"ru"` или `"pt-BR"`). Сервер нормализует значение через `resolveLocale()` — функцию из `src/config/locales.ts`, которая сначала ищет точное совпадение, затем сравнивает языковой префикс до дефиса, и при отсутствии совпадения молча откатывается на `en`. На основе разрешённой локали выбирается один из заранее загруженных HTML-шаблонов (`auth-code.en.html` или `auth-code.ru.html`), а тема письма подставляется из встроенной таблицы переводов. Все шаблоны загружаются в память при старте — если файл шаблона отсутствует, приложение не поднимется.

Каждый шаблон содержит magic link вида `{APP_BASE_URL}/deeplink-auth?code=<code>` для входа в один клик и тот же код в явном виде для ручного ввода. Срок действия в письме указан как 15 минут.

## Жизненный цикл токена

После получения JWT все защищённые ресурсы используют `Bearer`-аутентификацию через `JwtAuthGuard`. Выход осуществляется через `POST /auth/logout` — токен записывается в таблицу `jwt_blacklist` и перестаёт быть действительным немедленно, не дожидаясь истечения срока. Blacklist очищается по отдельному cron-у.
